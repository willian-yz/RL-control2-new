# RL + Fluent 统一框架使用说明与注意事项

本文档对应当前代码结构：

- `rl_fluent_framework.py`：统一环境/训练框架
- `case 3 no sin.py`：Case3 入口脚本（单动作、常值速度）
- `case 4 no sin.py`：Case4 入口脚本（双动作、正弦速度）

---

## 1. 框架目标与结构

当前版本将 Case3 / Case4 放在同一套主逻辑中，仅通过配置区分：

1. **环境配置（EnvConfig）**：Fluent 工况与求解运行参数
2. **强化学习配置（RLConfig）**：动作维度、动作范围、速度表达式形式、PPO超参数
3. **训练/测试管理（ExperimentManager）**：训练、续训、测试、绘图、模型与元数据保存

这样可以保证：

- 主流程一致，便于论文与汇报描述
- case 差异清晰，不需要复制粘贴大段代码
- 后续扩展到 case5/case6 时，只新增配置和入口脚本

---

## 2. 文件说明

## 2.1 `rl_fluent_framework.py`

核心类：

- `EnvConfig`：
  - `case_id`：用于命名模型和历史文件
  - `cas_path` / `data_path` / `workdir`
  - `slice_len` / `max_decisions` / `processor_count`
  - `initial_tploss`：reset 初始损失兜底值
- `RLConfig`：
  - `action_dim`：1（仅A）或2（A+f）
  - `amplitude_range`、`frequency_range`
  - `expr_mode`：`constant` 或 `sin`
  - PPO 超参数
- `CompressorEnv`：
  - `reset`：读初场、设置初始边界，初始化 `last_tploss`
  - `step`：动作映射→边界设置→Fluent推进→读报告→奖励计算→写历史
- `HistoryRecorder`：按 mode（train/test）分文件追加写 CSV
- `ExperimentManager`：
  - `train`：支持从已有模型续训并累加步数
  - `test`：独立测试流程
  - `plot_history`：绘制 reward/tploss/A/f 曲线

## 2.2 `case 3 no sin.py` / `case 4 no sin.py`

两个文件现在只做三件事：

1. 构造本 case 的 `EnvConfig` / `RLConfig`
2. 调用 `manager.train(...)`
3. 调用 `manager.test(...)` + `manager.plot_history(...)`

---

## 3. 快速开始

> 先确保 Fluent 可用、许可证正常、并且 `cas/dat` 路径有效。

### 3.1 运行 Case3

```bash
python "case 3 no sin.py"
```

### 3.2 运行 Case4

```bash
python "case 4 no sin.py"
```

运行后会生成：

- 模型：`artifacts/my_model_case*_step*`
- 模型元数据：`artifacts/case*_model_meta.json`
- 训练历史：`history/case*_train_history.csv`
- 测试历史：`history/case*_test_history.csv`
- 曲线图：`artifacts/case*_train_history.png`、`artifacts/case*_test_history.png`

---

## 4. 续训与模型命名规则

模型命名规则：

- `my_model_case{case_id}_step{total_steps}`

其中：

- 如果是从头训练，`total_steps = train_steps`
- 如果是读取旧模型再训练，`total_steps = 旧模型步数 + train_steps`

示例（伪代码）：

```python
model_path = manager.train(
    train_steps=2000,
    load_model_path="artifacts/my_model_case4_step1000.zip",
)
# 保存后名称应为 my_model_case4_step3000
```

## 5. 历史记录说明（可续写）

每个 step 会记录：

- `reward`
- `tploss`
- `action_a` / `action_f`
- 物理量 `A` / `f`
- 当前边界表达式 `expr`

记录按模式分开：

- 训练写入 `case*_train_history.csv`
- 测试写入 `case*_test_history.csv`

这两个文件默认以**追加模式**写入，便于多次实验持续积累。

> 若希望每次实验从空白历史开始，请先手动备份或删除对应 CSV。

---

## 6. reset 与 tploss 读取机制

- `reset()` 时会设置 `last_tploss = initial_tploss`
- `step()` 时优先从最新 report 文件读取 tploss
- 若 report 文件缺失或暂时不可用，则回退使用 `last_tploss`，避免直接崩溃

因此建议：

1. `initial_tploss` 填一个合理的损失初值（与当前工况尺度一致）
2. 确保 Fluent 报告定义与输出路径稳定，减少回退触发

---

## 7. 如何新增一个新 case（推荐流程）

1. 复制一个入口脚本（例如复制 `case 4 no sin.py`）
2. 修改 `build_case*_configs()`：
   - `case_id`
   - `cas_path` / `data_path` / `workdir`
   - `action_dim`（1或2）
   - `expr_mode`（`constant` 或 `sin`）
   - 动作范围、baseline、训练超参数
3. 保持 `main()` 的训练/测试/绘图流程不变

## 8. 常见问题与注意事项

## 8.1 路径问题

- 代码中 `workdir` 目前是绝对路径（Windows 风格）。
- 在不同机器运行时必须改成对应机器可访问路径。

## 8.2 Fluent 报告文件匹配

- 框架默认匹配前缀 `report-def-0-rfile` 的最新文件。
- 若你的报告命名不同，请修改 `EnvConfig.report_file_prefix`。

## 8.3 Gym / SB3 接口兼容性

- 目前环境采用 Gym 经典接口：`step -> obs, reward, done, info`。
- 如果未来升级到 Gymnasium，需要同步调整 API（如 terminated/truncated）。

## 8.4 历史文件并发写入

- 当前设计面向单进程顺序执行。
- 如果未来启用并行环境，建议为不同 worker 分离 history 文件名，避免写冲突。

## 8.5 可视化绘图依赖

- 需要 `matplotlib`。
- 无图形环境下建议使用非交互后端（例如 `Agg`），避免保存图片时报错。

---

## 9. 建议的科研使用习惯

1. 每次实验固定 `case_id + 配置快照`（可保存 `meta.json`）
2. 训练前清理或归档旧 history，防止不同实验混写
3. 统一记录：
   - 使用的模型起点（是否续训）
   - 本次 `train_steps`
   - Fluent 版本、网格版本、UDF版本
4. 论文图建议优先使用 `history csv` 二次加工作图（可复现）

---
